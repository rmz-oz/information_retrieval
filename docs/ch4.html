4.1 Hardware Basics

All index construction decisions are shaped by hardware constraints — particularly the gap between memory speed and disk speed. The Reuters-RCV1 collection (~800K docs, 100M tokens) requires 0.8 GB just for termID–docID pairs, making in-memory sorting impossible for large collections.

Key Hardware Principles

Seek Time Dominates

Moving the disk head averages 5 ms. Reading 10 MB as one contiguous chunk: 1 seek + transfer = 0.005 + 0.2 = 0.205 s. Reading the same 10 MB in 100 non-contiguous chunks: 100 seeks + transfer = 0.5 + 0.2 = 0.7 s (~3.4× slower).

Block-Level I/O

OS reads/writes entire blocks (8–64 KB). Reading one byte costs as much as a full block. Use a buffer — a region of main memory holding a block being read/written. Data transfers run via the system bus (not CPU), so they can overlap with computation.
Compression Pays Off Storing compressed data on disk is often faster overall because transfer time (b) is the bottleneck, not decompression CPU cost. Compressed transfers + fast decompression beats uncompressed transfers.

4.2 Blocked Sort-Based Indexing (BSBI)

BSBI segments the collection into blocks of equal size, sorts each block in memory, writes it to disk, then merges all blocks into a final index. Terms are represented as termIDs (integers) to save memory.

BSBI Algorithm Steps

StepAction
1Initialize block counter n = 0
2–3While unprocessed documents exist, increment n
4PARSENEXTBLOCK(): parse documents into (termID, docID) pairs until block is full
5BSBI-INVERT(block): (i) sort pairs by termID then docID; (ii) group docIDs per termID into postings lists — this is inversion
6WRITEBLOCKTODISK(block, f_n): write inverted block to disk file f_n
7MERGEBLOCKS(f_1,...,f_n; f_merged): simultaneously merge all block files using a priority queue (min-heap)

Inversion

Two steps: (1) sort all (termID, docID) pairs by termID then docID; (2) collect all docIDs per termID into a postings list.

Merging

All block files opened simultaneously. A min-heap always selects the globally smallest termID. All postings for that termID are merged and written to the final index file.

Reuters-RCV1

10M pairs per block → 10 blocks. Each block = 80 MB. Total construction ≈ 100 s.
Time Complexity: O(T log T) — sorting T termID–docID pairs dominates. In practice, PARSENEXTBLOCK (text parsing + I/O) and MERGEBLOCKS (sequential disk reads) are the dominant costs, not the in-memory sorts.

4.3 Single-Pass In-Memory Indexing (SPIMI)

BSBI requires a global term-to-termID mapping which may not fit in memory for very large collections. SPIMI eliminates this by using terms directly instead of termIDs.

Key SPIMI Innovations

1 — No TermID Mapping

Terms stored directly in a hash-based dictionary. No global term-to-ID table required → blocks can be much larger.

2 — Dynamic Postings Lists

Each posting is added directly to the term's postings list as tokens arrive. When a list fills up, it is doubled in size. No sorting of pairs needed.

SPIMI-INVERT (per block)

for each token (term, docID):
  if term not in dictionary:
    dictionary[term] = new postings list    # line 6
  else:
    postings = get_postings(term)          # line 7
  if postings list is full:
    postings = double_list(postings)       # lines 8-9
  append docID to postings                 # line 10
sort(dictionary.keys())                    # line 11 — only terms, not pairs
write block to disk                        # line 12
Time Complexity: O(T) — each token processed in O(1) amortized time (postings list doubling). Only the terms are sorted before writing (O(Mblock log Mblock) where Mblock ≪ T). Compression of both postings and dictionary is built-in.

4.4 Distributed Indexing with MapReduce

For web-scale collections, index construction must be distributed across clusters. MapReduce is a general distributed computing architecture for large clusters of commodity machines. Key fault-tolerance idea: any machine can fail; the master node reassigns its work automatically.

Map Phase — Parsers

Each parser processes one split and emits (termID, docID) key-value pairs. Output is written to local disk as segment files — one per term partition (e.g., a–f, g–p, q–z for j=3 partitions). Term partitions are defined by key ranges.

Reduce Phase — Inverters

Each inverter handles one term partition, reading r segment files (one per parser). It collects all docIDs for each termID, sorts them, and writes the final postings list. Each segment file requires only one sequential read, minimizing network traffic.

MapReduce Schema

map: web collection → list(termID, docID)

reduce: (termID, list(docID)) → postings list

Why Dynamic Split Assignment?

The master assigns splits dynamically (not pre-assigned). If a machine fails or is slow, its split is simply reassigned to another available machine — fault tolerance at no extra cost.
Load Balancing Problem (Zipf's Law) Term frequencies follow a power law — alphabetic partitioning (a–f, g–p, q–z) creates severe imbalance ('the', 'a', 'in' dominate certain partitions). Solution: sampling-based partitioning — sample 1% of the collection first, compute term-frequency distribution, then choose partition boundaries for equal total postings.

4.5 Dynamic Indexing

Real collections change constantly: new documents arrive, old ones are deleted or updated. A static index must be rebuilt periodically (acceptable for static collections like the Bible, but not for live web indexes).

Auxiliary Index Approach

Maintain two indexes: a large main index on disk + a small auxiliary index in memory for new documents. Searches run across both. Deletions handled with an invalidation bit vector — deleted documents filtered before results are returned. Updates = delete + re-insert.

One File per Postings List

Merging is easy (extend each list), but most file systems cannot handle very large numbers of small files efficiently.

One Large File (Concatenated)

Simpler, but updating requires rewriting the entire file. Each document update can cost up to Mave disk seeks.

Logarithmic Merging

To avoid the O(T²/n) cost of simple auxiliary merging, use logarithmic merging. Maintain indexes I0, I1, I2, ... on disk of sizes n, 2n, 4n, ... The in-memory index Z0 holds up to n postings.

Algorithm: When Z0 is full → transfer to I0 (if I0 empty). If I0 exists → merge Z0 + I0 → Z1 of size 2n, then check I1, and so on. Each posting is processed once per level, 'percolating up' through at most log2(T/n) levels.

Simple Aux Merging

O(T²/n) total cost — every new batch rewrites the growing main index.

Logarithmic Merging

O(T log(T/n)) — each posting merged at most log(T/n) times. Much better.

Trade-offs

Query must merge log(T/n) indexes (not just 2). IDF statistics become approximate. Some large engines rebuild from scratch periodically.

4.6 Other Index Types

Impact-Sorted vs. DocID-Sorted Postings

DocID-Sorted

Postings ordered by increasing docID. Advantages: (1) efficient gap encoding compression (small differences); (2) new documents always appended at the end — simple dynamic updates.

Impact-Sorted

Postings ordered by decreasing weight (e.g., tf-idf). Advantage: early termination — once weights are too small to change the top-k result, scanning stops. Complication: new documents may need insertion anywhere in the list → breaks append-only updates.

Security and Access Control Lists (ACLs)

In enterprise IR, a low-level employee should not find salary documents in search results. Access Control Lists solve this by inverting the user-document permission matrix.

  • Represent each document as the set of users who can access it.
  • Invert this matrix: for each user, a 'postings list' of document IDs they can access.
  • At query time: intersect text-based results with the user's ACL postings list.
Security + Spelling Correction Interaction Spelling correction models built from all documents (including restricted ones) can leak information. Example: if a restricted document contains 'ProjectXenon', an unauthorized user querying a typo like 'ProjectXenon' might receive a 'Did you mean?' suggestion — confirming the term exists in the corpus. Solution: build separate correction models per access level.

Q1 — Hardware: Quantify the cost of fragmented vs. contiguous disk storage for BSBI. ▼

Reading 10 MB as one contiguous chunk: 1 seek + transfer = (5×10−3) + (10×106 × 2×10−8) = 0.005 + 0.2 = 0.205 s

Reading the same 10 MB as 100 non-contiguous chunks: 100 seeks + transfer = 100 × 0.005 + 0.2 = 0.5 + 0.2 = 0.7 s

The fragmented case is ~3.4× slower. For index merging with millions of seeks, this difference is catastrophic.

Why BSBI minimizes seeks: BSBI writes each block as a single sequential file and reads it back sequentially during merging. The only seeks are one per block file open — not one per record. This is why external sorting algorithms using sequential I/O are orders of magnitude faster than naive disk-based sorting (Exercise 4.1 shows naive disk sort of Reuters-RCV1 would take ~308 days).

Q2 — BSBI vs. SPIMI: Explain why SPIMI achieves O(T) while BSBI is O(T log T). ▼

BSBI is O(T log T) because it must sort all T (termID, docID) pairs. Sorting T items requires O(T log T) comparisons across all blocks. This sort is the dominant step in theory.

SPIMI's two key innovations eliminate sorting of pairs:

  1. No termID mapping: Uses terms directly (hash table dictionary), eliminating the global term-to-termID table that may not fit in memory. Blocks can be much larger.
  2. Dynamic postings lists: Each incoming token is appended directly to its term's postings list (doubled when full). No collection of all pairs followed by sorting — only the terms (not the pairs) are sorted before writing to disk.

Term-sort per block is O(Mblock log Mblock) where Mblock ≪ T. Each token is processed in O(1) amortized time (hash lookup + list append). Total: O(T).

Q3 — MapReduce: Explain splits, parsers, segment files, and inverters. Why dynamic split assignment? ▼

Splits: The input collection is divided into n chunks of 16–64 MB. The master assigns splits dynamically — as a machine finishes one split, it gets the next. This provides automatic fault tolerance: if a machine fails, its split is simply reassigned.

Parsers (map phase): Each parser processes one split and emits (termID, docID) key-value pairs. Output is written to local disk as j segment files, one per term partition (e.g., a–f, g–p, q–z).

Segment files: Each parser creates j files. The master tells each inverter the locations of its r segment files (one per parser). Each segment file requires only one sequential read.

Inverters (reduce phase): Each inverter handles one term partition, reads r segment files, collects all docIDs for each termID, sorts them, and writes the final postings list.

Why dynamic assignment? Pre-assigning splits would mean a slow or failed machine blocks progress. Dynamic assignment ensures the cluster always works at full capacity — any available machine takes the next available split.

Q4 — Logarithmic Merging: Describe the algorithm and prove O(T log(T/n)) complexity. ▼

Algorithm: Maintain on-disk indexes I0, I1, ..., Ik of sizes n, 2n, 4n, ..., 2kn. In-memory Z0 holds up to n postings.

When Z0 fills up:

  • If I0 is empty: write Z0 to I0.
  • If I0 exists: merge Z0 + I0 → Z1 of size 2n. Then check I1. If I1 empty, write Z1 to I1. If I1 exists, merge → Z2. And so on.

State follows binary counting: The active index pattern equals the binary representation of k = T/n (the number of Z0 fills). Bit i = 1 means Ii is in use.

Complexity proof: Each posting starts at level 0. Each time it is involved in a merge, it moves up one level. There are at most log2(T/n) levels. So each posting is processed at most log2(T/n) times. Total work = T × log(T/n) = O(T log(T/n)).

Trade-offs: Query must now merge results from log(T/n) indexes (not just 2). IDF statistics are approximate (auxiliary index documents not in main index). Some large engines rebuild from scratch periodically to avoid these complexities.

Q5 — IDF Error: Show how a small auxiliary index causes large idf error for a trending term. ▼

Setup: Main index: Nmain = 800,000 documents. Auxiliary index: Naux = 1,000 new documents. Total N = 801,000.

Scenario — Tropical Storm 'Flossie':

Before the storm: 'Flossie' appears in dfmain = 2 documents.
idf (main only) = log(800,000 / 2) = log(400,000) ≈ 12.9

After the storm: 500 new articles about Flossie arrive in the auxiliary index (dfaux = 500).
True idf = log(801,000 / 502) = log(1,596) ≈ 7.37

Error: System computes idf = 12.9; true value = 7.37. Difference = 5.53 → 75% overestimate.

This means 'Flossie' would be scored as an extremely rare, distinctive term when in reality it has become a high-frequency term. Any query about Flossie would over-rank documents.

Why the error is large: Flossie's df jumps 250× (from 2 to 502). Because idf uses a logarithm, this maps to a large idf change. Even a tiny auxiliary index (1,000 out of 801,000 = 0.12% of total) causes catastrophic idf error for rapidly trending terms.
