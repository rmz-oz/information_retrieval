5.1 Statistical Properties of Terms

Chapter 5 compresses the two main components of an inverted index: the dictionary and the postings file. All techniques here are lossless — the original data can be fully recovered. Compression serves three goals: less disk space, better cache utilization (fit more in RAM), and faster disk-to-memory transfer.

Lossless vs. Lossy Lossless: VB codes, γ codes, blocked dictionary, front coding — index answers every query identically. Lossy: case folding, stemming, stop-word removal (covered in Ch. 2) — some information is permanently discarded.

  • Stop word removal (150 words) cuts postings by 30–39% but barely touches vocabulary — stop words are few but extremely frequent.
  • Rule of 30: The 30 most common words account for 30% of all tokens (31% in RCV1).
  • Compression beats stop-word removal: VB encoding gives 116 MB vs. the stop-word-only uncompressed index at ~255 MB.

5.1.1 Heaps' Law — Estimating Vocabulary Size

Heaps' law estimates vocabulary size M as a function of collection size T (number of tokens):

M = k · Tb    where 30 ≤ k ≤ 100 and b ≈ 0.5
For Reuters-RCV1: k = 44, b = 0.49 → M = 44 × T0.49
Verification: T = 1,000,020 → M = 44 × 1,000,0200.49 ≈ 38,323. Actual: 38,365. ✓

Vocabulary Never Stops Growing

There is no maximum vocabulary size for a large enough collection. We cannot pre-allocate a fixed dictionary.

M Is Very Large

For large collections M grows as T0.5. This makes dictionary compression essential, not optional.

5.1.2 Zipf's Law — Distribution of Term Frequencies

Zipf's law models the collection frequency (cf) of terms ranked by frequency:

cfi ∝ 1/i (rank i has frequency ∝ 1/i) — a power law with exponent −1
log cfi = log c + k·log i (k = −1) → linear on a log-log plot
For RCV1 (M = 400,000): HM ≈ ln(400,000) ≈ 13 → c = 1/13. Term i occurs ≈ 15/i times per document.

Zipf's law explains why gap encoding works well for frequent terms (gaps ≈ 1) but less well for rare terms (gaps ≈ docID magnitude — no savings). It also justifies variable encoding: the gap distribution is heavily skewed toward small values.

5.2 Dictionary Compression

The dictionary is small relative to the postings file, but fits in RAM — keeping it compressed means fewer disk seeks during query processing. For RCV1, assume M = 400,000 terms with an average of 8 characters each.

5.2.1 Fixed-Width Array

Allocate a fixed slot per term: 20 bytes (term) + 4 bytes (doc freq) + 4 bytes (postings pointer) = 28 bytes/entry.

RCV1: 400,000 × 28 = 11.2 MB. Binary search over sorted array.
Problems
  • Wastes ~12 bytes/term on average (avg English term ≈ 8 chars, not 20).
  • Cannot store terms longer than 20 chars (e.g., hydrochlorofluorocarbons).

5.2.2 Dictionary as a String

Store all terms concatenated in one long string. Per entry: 4 (freq) + 4 (postings ptr) + 3 (term ptr into string) + avg 8 (term chars) = 19 bytes.

RCV1: 400,000 × (4+4+3) + 400,000 × 8 = 4.4 MB + 3.2 MB = 7.6 MB
Savings: 11.2 − 7.6 = 3.6 MB (32% reduction)
Term pointers need log2(3.2 × 106) ≈ 22 bits ≈ 3 bytes.

5.2.3 Blocked Storage

Group consecutive terms into blocks of size k. Store only 1 term pointer per block (the first term). Add 1 byte per term for its string length. For k = 4:

Savings per block

Eliminate (k−1) = 3 term pointers × 3 bytes = 9 bytes saved

Overhead per block

Add k = 4 length bytes → 4 bytes added
Net per block = 9 − 4 = 5 bytes. Total: 400,000/4 × 5 = 0.5 MB saved.
Result: 7.6 − 0.5 = 7.1 MB. Trade-off: binary search over blocks + linear scan within block.

Front Coding

Alphabetically adjacent terms often share prefixes. Front coding encodes only the difference from the shared prefix. Mark the common prefix end with *, then encode the varying suffix.

automata, automate, automatic, automation
→ 8automat*a1■e2■ic3■ion
  (8 = length of "automata"; 1,2,3 = length of varying suffix)
Applied on top of blocked storage for RCV1: saves an additional 1.2 MB → 5.9 MB.
Perfect Hashing (mapping M terms to [1..M] without collisions) gives even better compression but cannot be used in dynamic environments — adding any new term causes a collision and requires rebuilding the entire hash function.

5.3 Postings File Compression

RCV1: 800,000 docs → docIDs need log2(800,000) ≈ 20 bits each. Uncompressed: 100M × 20/8 = 250 MB. Using 32-bit integers: 400 MB. Goal: compress this significantly.

Gap Encoding

Instead of storing absolute docIDs, store the gaps (differences between consecutive docIDs). Frequent terms have gaps ≈ 1; rare terms have gaps ≈ docID magnitude.

We need a variable encoding that uses few bits for small gaps and more bits for large gaps.

5.3.1 Variable Byte (VB) Codes

Each byte has a 7-bit payload + 1-bit continuation flag (MSB). If the continuation bit = 0: more bytes follow. If = 1: this is the last byte of this gap.

Encoding Algorithm

Repeatedly extract 7-bit chunks from the gap (LSB first). Set continuation bit to 1 on the final byte, 0 on all preceding bytes.

Decoding Algorithm

Accumulate 7-bit payloads until encountering a byte whose continuation bit = 1. That completes one gap value.

5.3.2 γ (Gamma) Codes

For even better compression at the cost of more complex decoding, use bit-level codes. γ encoding splits a gap G into two parts:

  • Offset: G in binary with the leading 1 removed. For G=13=1101₂, offset = 101.
  • Length: the bit-length of the offset (= ⌊log2G⌋), encoded in unary. For length=3, unary = 1110.
  • γ code = length (unary) + offset. For G=13: γ = 1110 101 = 1110101
Total bit length = 2⌊log2G⌋ + 1 (always odd). γ codes are always within a factor of 2 of the theoretically optimal code length log2G.

Key Properties of γ Codes

Universal

Within a factor of 3 of optimal for any probability distribution P over gaps: E(Lγ)/H(P) ≤ 3 for H(P) > 1.

Prefix-Free

No γ code is a prefix of another. Enables unique decoding without delimiters — just scan bits left-to-right.

Parameter-Free

Requires no model fitting. Other codes (Golomb, Bernoulli-optimal) require fitting to the actual gap distribution, which changes dynamically.
γ encoding of RCV1 postings: 101 MB (predicted 224 MB via Zipf analysis; actual is better because Zipf's law is approximate and gaps are non-uniform within blocks).

δ (Delta) Codes

Like γ codes but encode the length part using γ code instead of unary. For G with offset length L: δ(G) = γ(L) + offset. More efficient for large G because γ(L) is shorter than unary(L) for large L.

Example: δ(1025): G=1025, offset has 10 bits. γ(10) = 1110010 (7 bits). δ(1025) = 1110010,0000000001 = 17 bits vs γ(1025) = 21 bits. Savings = 4 bits.
δ codes are better than γ for large gaps (rare terms, web-scale collections).

VB vs. γ Trade-off & Full Size Table

Use γ codes when…

Disk space is the scarce resource. Compression ratio is paramount. Accept slower decoding (~15% better than VB).

Use VB codes when…

Query throughput matters more than space. Byte-aligned → no bit manipulation → fast decoding. Most practical IR systems choose VB.

Question 1
What are the three main benefits of compressing an IR index?
Compression reduces disk space, fits more index in RAM (cache utilization), and allows faster transfer because decompression is cheap — a small compressed file transfers faster than a large uncompressed one.
Question 2
Heaps' law is M = kTb. What are the typical parameter ranges and the RCV1 best-fit values?
Standard ranges: 30 ≤ k ≤ 100, b ≈ 0.5. RCV1 best fit: k=44, b=0.49, giving M = 44 × T0.49. Verification: T=1,000,020 → predicted 38,323, actual 38,365.
Question 3
What does Heaps' law imply about vocabulary growth as the collection grows?
Because b < 1, M grows without bound as T increases — just more slowly. We cannot pre-allocate a fixed dictionary size. This is the key implication that makes dictionary compression essential.
Question 4
Zipf's law states that the collection frequency of the term ranked i is proportional to:
Zipf's law: cfi ∝ 1/i. This is a power law with exponent −1 (linear on a log-log plot). It means the most frequent term is twice as frequent as the second, three times the third, etc.
Question 5
Why does gap encoding benefit frequent terms much more than rare terms?
Zipf's law explains this: 'the' (rank 1) appears in most documents so gaps ≈ 1, encodable in a handful of bits. 'arachnocentric' (rare) appears in only 2 documents; its gap ≈ 248,100, requiring nearly as many bits as a raw docID.
Question 6
For Reuters-RCV1 with M=400,000 terms, what is the size of the fixed-width dictionary?
Fixed-width: 20 bytes (term) + 4 bytes (doc freq) + 4 bytes (postings ptr) = 28 bytes/entry. 400,000 × 28 = 11.2 MB.
Question 7
In the dictionary-as-a-string approach, how many bytes does each term's pointer into the string require for RCV1?
The term string for 400,000 terms × 8 chars = 3.2 MB = 3.2 × 106 positions. log2(3.2 × 106) ≈ 22 bits ≈ 3 bytes. Total per entry: 4+4+3 = 11 bytes overhead + 8 chars = 19 bytes → 7.6 MB total.
Question 8
In blocked dictionary storage with k=4, what is the net saving per block (in bytes)?
Save: (k−1) × 3 = 3 × 3 = 9 bytes (eliminated term pointers). Add: k × 1 = 4 bytes (length bytes). Net = 9 − 4 = 5 bytes per block. For RCV1: 400,000/4 × 5 = 0.5 MB saved → 7.1 MB total.
Question 9
What does the front coding of 'automata', 'automate', 'automatic', 'automation' look like?
'8automat*a1■e2■ic3■ion': 8 = length of 'automata' (the first term in full), 'automat' is the common prefix (marked with *), then each subsequent term gives its suffix length + suffix: 1+a, 2+ic, 3+ion (from 'automate'→'e', 'automatic'→'ic', 'automation'→'ion').
Question 10
In VB encoding, what is the role of the continuation bit and what is the maximum gap encodable in 2 bytes?
Continuation bit = 1 → last byte (stop reading). Two bytes give 14 payload bits → max = 214 − 1 = 16,383. One byte: 27−1 = 127. Three bytes: 221−1 = 2,097,151.
Question 11
What is the γ code for G = 24?
G=24: binary = 11000 (5 bits). Leading 1 removed → offset = 1000 (4 bits). Length = 4. Unary(4) = 11110. γ(24) = 11110,1000 = 9 bits. Formula check: 2×⌊log224⌋+1 = 2×4+1 = 9. ✓
Question 12
Which of the following is NOT a property of γ codes?
γ codes are bit-level codes — boundaries fall in the middle of machine words, requiring shifts and masks. This is precisely why γ decoding is slower than VB. VB codes are byte-aligned, not γ codes.
Question 13
What are the actual postings file sizes for RCV1 with VB and γ encoding respectively?
VB = 116 MB, γ = 101 MB (γ is ~15% better). Both are far below the 250 MB uncompressed (20-bit) or 400 MB (32-bit) baselines. Note γ < VB — γ achieves better compression at the cost of slower decoding.
Question 14
How do δ (delta) codes differ from γ codes, and when are δ codes better?
δ(G) = γ(L) + offset where L is the offset length. For G=1025: γ(1025) needs unary(10)=11 bits for length → 21 bits total; δ(1025) uses γ(10)=7 bits for length → 17 bits total. δ wins for large G (rare terms in large collections).
Question 15
The compressed RCV1 index (γ-encoded postings + best dictionary) is approximately what fraction of the raw collection size?
(101 MB postings + 5.9 MB dictionary) / 3600 MB collection ≈ 0.03 = 3%. Even vs. text-only (960 MB) it is about 11%. This remarkable result shows that a compressed inverted index is far smaller than the collection it indexes.
1. Heaps' Law Prediction & Vocabulary Growth ›

Question: Using Heaps' law with k=44 and b=0.49 (RCV1 parameters), predict the vocabulary size for a collection of T = 10,000,000,000 (10 billion) tokens. What does this imply for IR system design?

Solution:

M = 44 × (1010)0.49 = 44 × 100.49 × 10 = 44 × 104.9

104.9 = 105 × 10−0.1 ≈ 100,000 / 1.259 ≈ 79,433

M = 44 × 79,433 ≈ 3,495,000 ≈ 3.5 million distinct terms

Implications:

  • The vocabulary is enormous — fixed-width storage at 28 bytes × 3.5M = 98 MB just for the dictionary. String-based at 19 bytes = 66.5 MB, still large.
  • The vocabulary will keep growing — any new document can introduce new terms. Systems must support dynamic dictionary expansion.
  • Front coding and blocked storage become essential, not optional.
  • The vocabulary never saturates, meaning we can never cache the full dictionary in L1/L2 cache — L3 and RAM-level caching strategies are needed.

Compare: RCV1 (T=1M) → M≈38K. Web-scale (T=10B) → M≈3.5M. A 10,000× larger collection only gives a ~92× larger vocabulary (T0.49 grows slowly), confirming the sub-linear nature of Heaps' law.

2. Dictionary Compression: Size Calculations Step by Step ›

Question: For RCV1 (M=400,000 terms, avg term length=8 chars), compute the dictionary size for: (a) fixed-width array, (b) dictionary-as-a-string, (c) blocked storage k=8, (d) with additional front coding saving 1.2 MB. Show all calculations.

Solution:

(a) Fixed-width array:

  • Per entry: 20 bytes (term) + 4 bytes (doc freq) + 4 bytes (postings ptr) = 28 bytes
  • Total: 400,000 × 28 = 11,200,000 bytes = 11.2 MB

(b) Dictionary as a string:

  • Term string total: 400,000 × 8 = 3,200,000 bytes = 3.2 MB
  • Term pointer per entry: log2(3.2×106) ≈ 22 bits → 3 bytes
  • Per entry: 4 (freq) + 4 (postings ptr) + 3 (term ptr) = 11 bytes overhead + 8 bytes (term chars in string)
  • Total: 400,000 × 11 + 3,200,000 = 4,400,000 + 3,200,000 = 7,600,000 bytes = 7.6 MB
  • Savings: 11.2 − 7.6 = 3.6 MB (32% reduction)

(c) Blocked storage k=8:

  • Save: (k−1) × 3 = 7 × 3 = 21 bytes per block (eliminated term pointers)
  • Add: k × 1 = 8 bytes per block (length bytes)
  • Net per block: 21 − 8 = 13 bytes. Blocks: 400,000/8 = 50,000
  • Total savings: 50,000 × 13 = 650,000 bytes = 0.65 MB
  • Size: 7.6 − 0.65 = 6.95 MB ≈ 6.95 MB
  • Alternatively: 400,000 × (4+4+3/8+1+8) = 400,000 × 17.375 = 6,950,000 bytes ✓

(d) With front coding:

  • Additional saving ≈ 1.2 MB
  • Size: 6.95 − 1.2 ≈ 5.75 MB (for k=8+front coding)
  • For k=4 + front coding: 7.1 − 1.2 = 5.9 MB (as given in the book for k=4)

5. VB vs. γ vs. δ: Comparison and When to Use Each ›

Question: Compare VB, γ, and δ codes on compression ratio, decoding speed, and practical applicability. Give actual sizes for RCV1. For the gap G=511, compute the γ code and δ code, and compare their lengths.

Solution — RCV1 sizes:

γ code for G=511:

  • Binary(511) = 111111111 (9 bits). Offset = 11111111 (8 bits). Length = 8.
  • Unary(8) = 111111110 (nine characters: eight 1s then a 0).
  • γ(511) = 111111110,11111111 = 17 bits

δ code for G=511:

  • L = 8. γ(8): binary(8)=1000, offset=000 (3 bits), length=3, unary(3)=1110. γ(8) = 1110,000 = 7 bits.
  • δ(511) = γ(8) + offset = 1110000,11111111 = 15 bits
  • δ saves 2 bits vs. γ for G=511. For G=1025: δ saves 4 bits.

Practical guidance:

  • VB codes: Byte-aligned → no bit manipulation → simple, fast. Most real-world IR systems (Lucene, etc.) use VB or a variant. Choose when query throughput > space savings.
  • γ codes: Best compression for collections with many frequent terms (small gaps dominate). Parameter-free, universal. Choose when storage is the bottleneck and queries are infrequent.
  • δ codes: Best for large gaps (rare terms, web-scale collections). Strictly better than γ for G≥3 in most cases. Choose for very large, skewed collections.
