Chapter 3: Dictionaries and Tolerant Retrieval

Manning et al.

3.1 Search Structures for Dictionaries

When processing a query, the IR system must look up each query term in the dictionary (the vocabulary data structure) to retrieve its postings list pointer. Two broad classes of solutions exist: hashing and search trees.

Hashing

Each term is hashed to an integer index; collisions are handled by auxiliary structures. Query-time lookup is O(1). However, hashing has critical weaknesses for IR:

No Prefix Queries

Terms sharing a common prefix (e.g., all terms starting with 'automat') hash to completely different values — no structural locality.

No Variant Lookup

Cannot efficiently support minor variant lookup (accented vs. unaccented characters).

Growth Problem

Vocabulary size grows dynamically; a hash function designed for current needs may not suffice as the collection grows.

Search Trees — B-trees

Search trees require a prescribed ordering on characters (A–Z for English). They allow enumeration of all terms with a given prefix in O(log M) time, where M is the vocabulary size. The most widely used search tree for IR dictionaries is the B-tree: every internal node has between a and b children (for fixed integers a, b).

Binary Search Tree

O(log M) comparisons but requires rebalancing on insertions and deletions.

B-tree

Self-balancing, fan-out of [a,b] children per node; height = O(logb M). Each disk read fetches a full node — efficient for disk-resident dictionaries.

B-tree Advantages

Unlike hashing, B-trees support prefix queries, range queries, and wildcard lookups.
Key Insight A B-tree can be viewed as collapsing multiple levels of a binary search tree into one disk block per node, which is particularly efficient for disk-resident dictionaries (each node = one disk block, so each level = one disk read).

3.2 Wildcard Queries

A wildcard query uses the * symbol to represent any sequence of characters. Users use wildcard queries when: (1) uncertain about spelling (S*dney for Sydney/Sidney), (2) seeking variant spellings (color/colour), (3) seeking stemming variants (judicial*), (4) unsure of a foreign word's rendition (Universit* Stuttgart).

Trailing Wildcard Queries (mon*)

A query ending in * is a trailing wildcard. Handle with a standard B-tree: walk the tree following the prefix characters (m, o, n), then enumerate all terms in the resulting subtree — the set W. Look up each term in W in the standard inverted index.

Cost: O(log M + |W|) for enumeration plus |W| standard postings lookups.

Leading Wildcard Queries (*mon)

A query starting with * is a leading wildcard. Use a reverse B-tree: a B-tree built on reversed terms. The term 'lemon' is stored as 'nomel'. A leading wildcard *mon is reversed to nom*, then looked up in the reverse B-tree to enumerate all terms ending in 'mon'.

General Wildcard with One * (se*mon)

Handled using BOTH the regular B-tree and the reverse B-tree: enumerate W = terms starting with 'se' (B-tree), enumerate R = terms ending with 'mon' (reverse B-tree), then take the intersection W ∩ R.

3.2.1 Permuterm Index

For general wildcard queries with a single *, the permuterm index provides an elegant alternative. The idea: append a special end-of-term symbol $ to each vocabulary term, then store all rotations of the augmented term in a B-tree, each pointing back to the original term.

Example: term "hello"
Augmented: hello$. Rotations (6 total): hello$, ello$h, llo$he, lo$hel, o$hell, $hello — each pointing back to 'hello'.

Limitation: Multiple *

Queries like fi*mo*er require post-filtering the permuterm results — enumerate from er$fi* prefix, then filter to keep only those containing 'mo'.

Space Cost

A term of length k generates k+1 rotations. The permuterm index is ~10x larger than the original vocabulary.

3.2.2 k-gram Index for Wildcard Queries

A k-gram is a contiguous sequence of k characters. The k-gram index uses $ as a start/end-of-term marker. For the term castle, the 3-grams are: $ca, cas, ast, stl, tle, le$. The k-gram dictionary maps each k-gram to vocabulary terms containing it; postings are stored in lexicographic order.

Answering wildcard query re*ve: Generate the Boolean query $re AND ve$, look it up in the 3-gram index, and intersect the postings lists. Terms like 'relive', 'remove', 'retrieve' will match.
Post-filtering is Required The k-gram Boolean query may return false positives. For query red*, the query $re AND red returns 'retired' (contains $re and red as 3-grams, but does NOT start with 'red'). Each candidate must be string-matched against the original wildcard pattern.

Advantage over Permuterm

Much smaller index (~3–10x smaller) since k-gram count per term is linear in term length, not equal to the number of rotations.

Dual Use

The k-gram index is also used for spelling correction (Section 3.3.4), making it doubly useful.

3.3 Spelling Correction

The goal of spelling correction is to retrieve documents for the intended query even when the query contains spelling errors. Users make errors either accidentally or because a term has no unambiguous spelling.

Two Core Principles

Nearest Correction

Among all plausible correct spellings, select the one with minimum distance (edit distance or k-gram overlap) to the misspelled query term.

More Common Term

When two candidates are equally close, prefer the one that is more frequent in the collection (higher document frequency) or more frequently queried (from query logs).

Modes of Exposing Correction to Users

  1. Always retrieve documents for both the original query and all spell-corrected versions.
  2. Correct only when the original term is not in the dictionary.
  3. Correct only when the original query returns fewer than a threshold number of results (e.g., <5).
  4. Present a "Did you mean X?" suggestion to the user.

Isolated-term vs. Context-sensitive Correction

Isolated-term Correction

Correct each query term independently. Works for 'carot' → 'carrot'. Fails for 'flew form Heathrow' where 'form' is correctly spelled but semantically wrong ('from' is intended).

Context-sensitive Correction

Consider all query terms together. Enumerate corrections of each term, combine into candidate phrases, evaluate each against the collection (biword frequency or query logs), select most plausible.

3.3.3 Edit Distance (Levenshtein Distance)

The edit distance between strings s1 and s2 is the minimum number of single-character edit operations needed to transform s1 into s2. Standard operations: (i) insert a character, (ii) delete a character, (iii) substitute one character for another.

EDITDISTANCE(s1, s2):
  m[i,0] = i  for all i   (delete i chars from s1)
  m[0,j] = j  for all j   (insert j chars into s1)
  for i=1 to |s1|:
    for j=1 to |s2|:
      m[i,j] = min(
        m[i-1,j-1] + (0 if s1[i]==s2[j] else 1),  # substitute/match
        m[i-1,j] + 1,   # delete from s1
        m[i,j-1] + 1    # insert into s1
      )
  return m[|s1|][|s2|]

Time Complexity

O(|s1| × |s2|) — quadratic in string lengths.

Space Complexity

O(|s1| × |s2|), reducible to O(min(|s1|,|s2|)) with rolling arrays.

Bound

edit_distance(s1, s2) ≤ max(|s1|, |s2|)
Scalability Problem For a vocabulary of size V, computing edit distance from the query to every vocabulary term costs O(V × |s|²) — too slow for large vocabularies. Heuristics are needed: only compare terms sharing the first letter, or use k-gram indexing to pre-filter candidates.

3.3.4 k-gram Index for Spelling Correction

The k-gram index retrieves vocabulary terms that share many k-grams with the query term q. These are likely to have low edit distance to q. The Jaccard coefficient measures overlap:

Jaccard(A, B) = |A ∩ B| / |A ∪ B|
where A = k-grams of q, B = k-grams of vocabulary term t. Range: 0 (no overlap) to 1 (identical sets).

On-the-fly computation while scanning postings:

  • |A ∩ B|: count how many k-gram postings for q contain term t.
  • |A ∪ B| = |A| + |B| − |A ∩ B|: |A| is known; |B| is pre-stored as t's k-gram count.
Example (query "bord", bigrams):
bord's bigrams: bo, or, rd (|A|=3).
border (|B|=6 bigrams: $b, bo, or, rd, de, er) → shared = {bo,or,rd} = 3. Jaccard = 3/(3+6−3) = 0.5
boardroom (|B|=8) → shared = {bo,rd} = 2. Jaccard = 2/(3+8−2) = 0.22

3.3.5 Context-Sensitive Spelling Correction

For queries like 'flew form Heathrow' where all individual terms are correctly spelled, isolated-term correction fails. Context-sensitive correction:

  1. Enumerate spelling corrections for each query term (using edit distance / k-grams).
  2. Generate candidate corrected phrases by substituting corrections at each position.
  3. Evaluate each candidate phrase against a biword index or query log frequency.
  4. Select the candidate phrase with the highest frequency / plausibility.
Combinatorial Explosion For n misspelled terms each with k corrections: kn candidate phrases. Heuristic pruning (retain only top-10 most common bigrams at each prefix step) is essential. For 'flew form Heathrow', 'flew from' is a common bigram → selected as correction.

3.4 Phonetic Correction — Soundex

Phonetic correction handles queries where the user knows the sound of a name but not its spelling (e.g., 'Herman' could be 'Hermann', 'Herrmann'). The approach: generate a 'phonetic hash' such that similarly-sounding terms hash to the same code. The classic algorithm is Soundex, originally developed for U.S. census records.

Soundex Algorithm (5 Steps)

StepRule
1Retain the first letter of the term unchanged.
2Convert vowels A, E, I, O, U and the letters H, W, Y to '0' (zero).
3Map consonants: B,F,P,V→1 · C,G,J,K,Q,S,X,Z→2 · D,T→3 · L→4 · M,N→5 · R→6
4Remove consecutive duplicate digits (e.g., 55 → 5).
5Remove all zeros; pad with trailing zeros if needed; return the first 4 characters (1 letter + 3 digits).
Example: Hermann → H650
H retained · e→0 · r→6 · m→5 · a→0 · n→5 · n→5 (dup)
Digits: 0,6,5,0,5,5 → remove dups: 0,6,5,0,5 → remove 0s: 6,5,5 → remove dup: 6,5 → pad → H650

Herman → also H650. So searching 'herman' retrieves 'Hermann', 'Herman', 'Herrmann', etc.

Soundex Limitations
  • First letter always retained: Tchaikovsky (T212) ≠ Chaikovsky (C212) despite being phonetically identical — different first letters always produce different codes.
  • Writing-system dependent: Chinese names in Wade-Giles vs. Pinyin may map differently (e.g., Peking vs. Beijing).
  • Works well for European names but fails in many other cases.

Q1 — Hashing vs. B-tree: Which structure supports prefix enumeration and why? ▼

Answer: Only the B-tree supports prefix enumeration. A B-tree stores terms in sorted lexicographic order, so all terms sharing a common prefix form a contiguous range in the tree — enumerable with a single traversal from the prefix node down through the subtree.

Hashing distributes keys uniformly by design: two terms with a common prefix (e.g., 'mon', 'monday', 'monkey') will hash to completely different locations with no structural locality. There is no way to enumerate a contiguous range of hash values corresponding to a prefix.

Time complexities:

  • Hashing: O(1) average for exact lookup (amortised).
  • B-tree: O(logb M) where b = max children per node, M = vocabulary size. For disk-resident dictionaries, each B-tree level = one disk I/O → height is small (3–4 levels for large vocabularies).

Dynamic insertion: Both support it; B-tree rebalances automatically. Hashing may require rehashing if the table grows beyond its hash function's range.

Q2 — Permuterm Index: Build entries for 'ring' and answer the query r*ng. ▼

Permuterm entries for 'ring':
Augmented term: ring$. Rotations (5 total, each pointing back to 'ring'):

ring$  →  ring
ing$r  →  ring
ng$ri  →  ring
g$rin  →  ring
$ring  →  ring

Answering wildcard query r*ng:
Rotate so * appears at the end: ng$r*. Lookup prefix ng$r in the permuterm B-tree. All rotations beginning with 'ng$r' correspond to original terms that begin with 'r' AND end with 'ng'. Examples: ring, rang, rung, spring (ends ng), sting, strong.

Trailing ri*: Rotate to $ri*. Lookup $ri → all terms starting with 'ri'.

Leading *ng: Rotate to ng$*. Lookup ng$ → all terms ending with 'ng'.

Space disadvantage: A term of length k generates k+1 rotations. Average English word ≈ 6 chars → 7 entries per word. The permuterm vocabulary is ~7–10x larger than the original vocabulary.

Q3 — Edit Distance: Compute Levenshtein distance between 'grad' and 'greed'. ▼

Dynamic Programming Matrix (s1='grad' rows, s2='greed' cols):

       ""   g   r   e   e   d
  ""  [ 0   1   2   3   4   5 ]
   g  [ 1   0   1   2   3   4 ]
   r  [ 2   1   0   1   2   3 ]
   a  [ 3   2   1   1   2   3 ]
   d  [ 4   3   2   2   2   2 ]

Edit distance = 2.

Optimal path: substitute 'a'→'e' (grad → gred, cost 1), then insert 'e' before 'd' (gred → greed, cost 1). Total: 2 operations.

Should the system suggest 'greed'? Edit distance 2 is relatively high for a 4-letter word (50% of characters changed). Additional factors to consider:

  1. Collection frequency: prefer more common terms — if 'greed' is much more frequent than 'grad', suggest it.
  2. Context: in an academic context, 'grad' (graduate) is likely the intended term.
  3. Mode: only suggest correction if 'grad' is not in the dictionary or returns 0 results.
Q4 — Jaccard Coefficient: Compute for 'bord' vs. border, lord, morbid, sordid. ▼

Query q = 'bord'. Bigrams (A): bo, or, rd → |A| = 3.

borderbo,or,rd,de,er (+$b)63 (bo,or,rd)3/(3+6−3) = 3/6 = 0.600
lordlo,or,rd32 (or,rd)2/(3+3−2) = 2/4 = 0.500

Recommended correction: border (Jaccard = 0.6, highest; edit distance bord→border = 2 insertions).

Why Jaccard over raw count? Raw count favors longer terms simply because they have more bigrams to match. Jaccard normalizes by the union size, penalizing terms with many extra bigrams that don't appear in the query — giving a fair comparison across terms of different lengths.

Q5 — Soundex & Context-Sensitive Correction: Apply Soundex to Thompson vs. Thomson; explain top-10 pruning guarantee. ▼

Thompson: T · h→0,o→0,m→5,p→1,s→2,o→0,n→5 → digits: 0,0,5,1,2,0,5 → remove consec dups (00→0): 0,5,1,2,0,5 → remove 0s: 5,1,2,5 → T512

Thomson: T · h→0,o→0,m→5,s→2,o→0,n→5 → digits: 0,0,5,2,0,5 → remove 0s: 5,2,5 → T525

Thompson (T512) ≠ Thomson (T525), yet they are pronounced nearly identically. This illustrates a key Soundex limitation: the omitted 'p' in Thomson causes a different digit sequence.

Is top-10 pruning guaranteed to find the globally best correction? No.

The top-10 most common 2-term prefixes may NOT include the prefix that leads to one of the top-10 3-term phrases. Example: if the 2-term prefix 'cached in' is ranked 11th (just outside top-10) but 'cached in the rye' is the most common 3-term phrase, we would miss it — 'cached in' was pruned.

Top-10 pruning is a greedy heuristic that optimizes locally at each phase but provides no formal global guarantee. The true optimal algorithm would use beam search or A* search with a proper admissible heuristic. In practice, the heuristic works well because the most common 2-term prefixes are likely to extend to common 3-term phrases.
